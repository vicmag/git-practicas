TDD nos obliga a pensar sobre el diseño de los componentes

El Desarrollo Dirigido por Pruebas (TDD) nos obliga a diseñar nuestro código de manera estructurada y modular. Al seguir el ciclo Rojo - Verde - Refactor, logramos componentes bien definidos y fáciles de mantener.

1. Nos obliga a definir claramente la interfaz

Antes de escribir código, debemos pensar en cómo se usará el componente. Esto nos obliga a definir la interfaz de nuestra clase antes de su implementación.

Ejemplo:

Queremos diseñar un servicio para calcular descuentos en productos. Antes de implementarlo, escribimos una prueba:


import static org.junit.jupiter.api.Assertions.*;
import org.junit.jupiter.api.Test;

public class DiscountServiceTest {

    @Test
    public void testApplyDiscount() {
        DiscountService discountService = new DiscountService();
        double discountedPrice = discountService.applyDiscount(100, 10);
        assertEquals(90, discountedPrice, 0.01);
    }
}

Al escribir la prueba, definimos claramente que DiscountService debe tener un método applyDiscount, que reciba un precio y un porcentaje de descuento.

2. Nos ayuda a escribir código más modular

Las pruebas unitarias se centran en pequeñas partes del código, lo que nos obliga a diseñar componentes desacoplados y reutilizables.

Ejemplo:

En lugar de mezclar la lógica de descuentos dentro de la clase Product, creamos un servicio independiente:


public class DiscountService {
    public double applyDiscount(double price, double percentage) {
        return price * (1 - percentage / 100);
    }
}

Este diseño modular permite que otros componentes reutilicen DiscountService sin depender de una implementación específica de Product.

3. Facilita el mantenimiento y la evolución del código

Las pruebas actúan como una documentación viviente. Si en el futuro cambiamos la implementación, las pruebas nos indicarán si seguimos cumpliendo con el diseño original.

Ejemplo:

Supongamos que ahora queremos que los descuentos se apliquen solo si el precio es mayor a 50. Podemos modificar la implementación sin temor a romper el sistema, porque las pruebas nos guiarán.


public class DiscountService {
    public double applyDiscount(double price, double percentage) {
        if (price <= 50) {
            return price; // No se aplica descuento
        }
        return price * (1 - percentage / 100);
    }
}

Después del cambio, ejecutamos las pruebas. Si alguna falla, sabemos exactamente qué parte del código fue afectada.

Conclusión

TDD nos obliga a diseñar código más estructurado y mantenible. Nos ayuda a definir interfaces claras, escribir código modular y asegurar que los cambios futuros no rompan la funcionalidad existente.

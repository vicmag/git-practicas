TDD y su Impacto en el Diseño SOLID

El Desarrollo Guiado por Pruebas (TDD) no solo mejora la calidad del código, sino que también promueve activamente la aplicación de los principios SOLID. Veamos cómo cada aspecto de TDD se relaciona con estos principios fundamentales del diseño orientado a objetos.

1. Single Responsibility Principle (SRP)

Conexión con TDD:

TDD nos obliga a pensar en pruebas unitarias enfocadas, lo que naturalmente lleva a clases con una sola responsabilidad.

Ejemplo en el DiscountService:
public class DiscountService {
    // Solo se encarga de aplicar descuentos
    public double applyDiscount(double price, double percentage) {
        return price * (1 - percentage / 100);
    }
}

La prueba unitaria enfocada valida que esta clase solo hace una cosa: calcular descuentos.

2. Open/Closed Principle (OCP)

Conexión con TDD:

La fase de refactorización en TDD nos lleva a diseñar código abierto para extensión pero cerrado para modificación.

Ejemplo de extensión:
public interface DiscountStrategy {
    double apply(double price);
}

public class PercentageDiscount implements DiscountStrategy {
    private double percentage;
    
    public PercentageDiscount(double percentage) {
        this.percentage = percentage;
    }
    
    @Override
    public double apply(double price) {
        return price * (1 - percentage / 100);
    }
}

// Nueva estrategia sin modificar el código existente
public class FixedAmountDiscount implements DiscountStrategy {
    private double amount;
    
    public FixedAmountDiscount(double amount) {
        this.amount = amount;
    }
    
    @Override
    public double apply(double price) {
        return price - amount;
    }
}

Las pruebas guían la creación de esta estructura extensible.

3. Liskov Substitution Principle (LSP)

Conexión con TDD:

Las pruebas unitarias actúan como contratos que deben cumplir todas las implementaciones.

Pruebas para DiscountStrategy:
@Test
void percentageDiscount_shouldApplyCorrectDiscount() {
    DiscountStrategy strategy = new PercentageDiscount(10);
    assertEquals(90, strategy.apply(100), 0.01);
}

@Test
void fixedAmountDiscount_shouldApplyCorrectDiscount() {
    DiscountStrategy strategy = new FixedAmountDiscount(15);
    assertEquals(85, strategy.apply(100), 0.01);
}

Ambas implementaciones pueden sustituirse manteniendo el comportamiento esperado.

4. Interface Segregation Principle (ISP)

Conexión con TDD:

Al escribir pruebas primero, definimos interfaces específicas para cada cliente.

Interfaces específicas:
public interface DiscountApplier {
    double applyDiscount(double price);
}

public interface DiscountValidator {
    boolean isValid(double price, double percentage);
}

// En lugar de una interfaz grande:
// public interface DiscountService {
//    double applyDiscount(...);
//    boolean validate(...);
//    // otros métodos no relacionados
// }

TDD nos lleva naturalmente a interfaces más pequeñas y enfocadas.

5. Dependency Inversion Principle (DIP)

Conexión con TDD:

Las pruebas unitarias requieren inyección de dependencias y uso de mocks, promoviendo este principio.

Ejemplo con inyección:
public class OrderProcessor {
    private final DiscountApplier discountApplier;
    
    public OrderProcessor(DiscountApplier discountApplier) {
        this.discountApplier = discountApplier;
    }
    
    public double processOrder(Order order) {
        return discountApplier.applyDiscount(order.getTotal());
    }
}

// Prueba unitaria
@Test
void processOrder_shouldApplyDiscount() {
    DiscountApplier mockApplier = mock(DiscountApplier.class);
    when(mockApplier.applyDiscount(anyDouble())).thenReturn(90.0);
    
    OrderProcessor processor = new OrderProcessor(mockApplier);
    Order testOrder = new Order(100.0);
    
    assertEquals(90.0, processor.processOrder(testOrder), 0.01);
}

El diseño permite sustituir fácilmente la implementación real por un mock para pruebas.

Conclusión: TDD como Catalizador de SOLID

La práctica de TDD no solo mejora la calidad del código mediante pruebas, sino que también actúa como una fuerza guía que nos lleva naturalmente hacia diseños que cumplen con los principios SOLID:

  • Pruebas enfocadas → Single Responsibility
  • Refactorización constante → Open/Closed
  • Contratos de prueba → Liskov Substitution
  • Necesidad de mockear → Interface Segregation
  • Aislamiento para pruebas → Dependency Inversion

Al adoptar TDD, los desarrolladores internalizan estos principios de diseño casi sin esfuerzo, creando sistemas más mantenibles y extensibles por naturaleza.

Relación entre las Pruebas de Fase Roja y los Principios SOLID

Las pruebas unitarias en la fase roja de TDD no solo validan el comportamiento esperado, sino que también guían el diseño del código hacia principios SOLID. Analicemos cómo cada prueba promueve estos principios.

1. Single Responsibility Principle (SRP)

Conexión con las pruebas:

Las pruebas enfocadas validan que cada método tenga una única responsabilidad.

Ejemplo en setMinimumStockLevel:
@Test
void whenSetMinimuStockLevel_LevelIsSaved() {
    // Configuración
    ProductRepository productRepository = mock(ProductRepository.class);
    ProductService productService = new ProductService(productRepository);
    Product product = new Product("Camiseta Azul");
    int minimumStockLevel = 15;

    // Ejecución
    productService.setMinimumStockLevel(product, minimumStockLevel);

    // Validación
    verify(productRepository).save(product);
    assertEquals(minimumStockLevel, product.getMinimumStockLevel());
}

Esta prueba valida que el método solo debe:

  • Establecer el nivel mínimo de stock
  • Guardar el producto
Si intentara hacer más cosas, la prueba sería más compleja.

2. Open/Closed Principle (OCP)

Conexión con las pruebas:

Las pruebas de validación preparan el código para extensión.

Pruebas de validación:
@Test
void whenSetMinimuStockLevelWithInvalidValue_thenThrowException() {
    ProductRepository productRepository = mock(ProductRepository.class);
    ProductService productService = new ProductService(productRepository);
    Product product = new Product("Camiseta Azul");
    int invalidMinimumStockLevel = -10;
    
    assertThrows(IllegalArgumentException.class, 
        () -> productService.setMinimumStockLevel(product, invalidMinimumStockLevel));
    
    verify(productRepository, never()).save(product);
}

Estas pruebas llevan a implementar validaciones de manera que nuevas reglas puedan añadirse sin modificar la lógica existente:

// Diseño extensible para validaciones
public void setMinimumStockLevel(Product product, int minimumStockLevel) {
    validateMinimumStockLevel(minimumStockLevel); // Punto de extensión
    product.setMinimumStockLevel(minimumStockLevel);
    saveProduct(product);
}

3. Liskov Substitution Principle (LSP)

Conexión con las pruebas:

Las pruebas definen contratos que deben cumplir todas las implementaciones.

Pruebas para saveProduct:
@Test
void whenSaveProductWithNegativeStock_thenThrowException() {
    ProductRepository productRepository = Mockito.mock(ProductRepository.class);
    ProductService productService = new ProductService(productRepository);
    Product invalidProduct = new Product("Camiseta", -10);
    
    assertThrows(IllegalArgumentException.class, () -> {
        productService.saveProduct(invalidProduct);
    });
    
    verify(productRepository, never()).save(any());
}

Cualquier implementación de ProductRepository debe cumplir con:

  • No guardar productos con stock negativo
  • Lanzar IllegalArgumentException en estos casos

4. Interface Segregation Principle (ISP)

Conexión con las pruebas:

Los mocks en las pruebas revelan interfaces demasiado grandes.

Análisis de ProductRepository:
// Uso en pruebas muestra qué métodos se necesitan realmente
verify(productRepository).save(product);
verify(productRepository, never()).save(any());

Si ProductRepository tuviera muchos métodos no usados, las pruebas nos mostrarían que necesitamos segregar la interfaz.

5. Dependency Inversion Principle (DIP)

Conexión con las pruebas:

El uso de mocks demuestra la inversión de dependencias.

Inyección de dependencias:
// En todas las pruebas:
ProductRepository productRepository = mock(ProductRepository.class);
ProductService productService = new ProductService(productRepository);

Este diseño:

  • Depende de la abstracción ProductRepository
  • Permite sustituir la implementación real por un mock
  • Facilita el testing y los cambios futuros

Conclusión: Las Pruebas Guían el Diseño SOLID

Las pruebas en fase roja no solo verifican comportamiento, sino que también:

  • Revelan responsabilidades únicas (SRP) mediante pruebas enfocadas
  • Preparan para extensión (OCP) con pruebas de validación
  • Definen contratos claros (LSP) que deben cumplir las implementaciones
  • Muestran interfaces ideales (ISP) a través de los mocks usados
  • Requieren inversión de dependencias (DIP) para poder testear

Al escribir pruebas primero, el diseño naturalmente evoluciona hacia estos principios fundamentales.

Pruebas Unitarias

Mejores Prácticas en Pruebas Unitarias

1. Nombrado de la Prueba

El nombre de la prueba debe ser descriptivo y seguir un patrón que indique qué se está probando y cuál es el resultado esperado. Un formato común es:

[NombreDelMétodo]_[CondiciónDePrueba]_[ResultadoEsperado]

2. Estructura de la Prueba: Arrange-Act-Assert

  • Arrange: Preparar el entorno de prueba (objetos, datos de entrada, etc.).
  • Act: Ejecutar el método o funcionalidad que se está probando.
  • Assert: Verificar que el resultado obtenido es el esperado.

3. Principios FIRST

  • Fast: Las pruebas deben ejecutarse rápidamente.
  • Independent: Las pruebas no deben depender de otras pruebas.
  • Repeatable: Las pruebas deben producir el mismo resultado en cualquier entorno.
  • Self-Validating: Las pruebas deben determinar automáticamente si pasan o fallan.
  • Timely: Las pruebas deben escribirse antes o durante el desarrollo del código.

4. Un ASSERT por Prueba

Cada prueba debe verificar una única condición. Esto hace que las pruebas sean más claras y fáciles de mantener.

5. Scope de la Prueba

Las pruebas unitarias deben probar solo métodos públicos, ya que los métodos privados son detalles de implementación que no deben exponerse directamente.

Ejemplo en Java con JUnit

Código del Servicio de Descuentos

public class DiscountService {
    public double applyDiscount(double price, double percentage) {
        if (price <= 0 || percentage < 0) {
            throw new IllegalArgumentException("Price and percentage must be positive.");
        }
        return price * (1 - percentage / 100);
    }
}

Prueba Unitaria con Mejores Prácticas

import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

class DiscountServiceTest {

    @Test
    void applyDiscount_ValidPriceAndPercentage_ReturnsDiscountedPrice() {
        // Arrange
        DiscountService discountService = new DiscountService();
        double price = 100;
        double percentage = 10;

        // Act
        double discountedPrice = discountService.applyDiscount(price, percentage);

        // Assert
        assertEquals(90, discountedPrice, 0.01, "The discounted price should be 90.");
    }

    @Test
    void applyDiscount_ZeroPrice_ThrowsIllegalArgumentException() {
        // Arrange
        DiscountService discountService = new DiscountService();
        double price = 0;
        double percentage = 10;

        // Act & Assert
        IllegalArgumentException exception = assertThrows(IllegalArgumentException.class, () -> {
            discountService.applyDiscount(price, percentage);
        });

        assertEquals("Price and percentage must be positive.", exception.getMessage());
    }

    @Test
    void applyDiscount_NegativePercentage_ThrowsIllegalArgumentException() {
        // Arrange
        DiscountService discountService = new DiscountService();
        double price = 100;
        double percentage = -10;

        // Act & Assert
        IllegalArgumentException exception = assertThrows(IllegalArgumentException.class, () -> {
            discountService.applyDiscount(price, percentage);
        });

        assertEquals("Price and percentage must be positive.", exception.getMessage());
    }
}

Explicación del Código de Prueba

1. Nombrado de la Prueba

  • applyDiscount_ValidPriceAndPercentage_ReturnsDiscountedPrice: Prueba que el método applyDiscount devuelve el precio correcto con un descuento válido.
  • applyDiscount_ZeroPrice_ThrowsIllegalArgumentException: Prueba que el método lanza una excepción cuando el precio es cero.
  • applyDiscount_NegativePercentage_ThrowsIllegalArgumentException: Prueba que el método lanza una excepción cuando el porcentaje es negativo.

2. Estructura Arrange-Act-Assert

  • Arrange: Se preparan los datos de entrada y se instancia el objeto DiscountService.
  • Act: Se ejecuta el método applyDiscount.
  • Assert: Se verifica el resultado esperado (ya sea un valor o una excepción).

3. Principios FIRST

  • Fast: Las pruebas se ejecutan rápidamente, ya que no dependen de recursos externos.
  • Independent: Cada prueba es independiente y no depende del resultado de otras pruebas.
  • Repeatable: Las pruebas producen el mismo resultado en cualquier entorno.
  • Self-Validating: Las pruebas determinan automáticamente si pasan o fallan mediante aserciones.
  • Timely: Las pruebas se escriben junto con el código que están probando.

4. Un ASSERT por Prueba

Cada prueba tiene un único assert para verificar una condición específica. Esto hace que las pruebas sean más claras y fáciles de mantener.

5. Scope de la Prueba

Las pruebas solo verifican métodos públicos (applyDiscount). No se prueban métodos privados, ya que son detalles de implementación.

Conclusión

Este ejemplo en Java con JUnit muestra cómo aplicar las mejores prácticas en pruebas unitarias:

  1. Nombrado descriptivo: Los nombres de las pruebas indican claramente qué se está probando.
  2. Estructura AAA: Las pruebas siguen el patrón Arrange-Act-Assert.
  3. Principios FIRST: Las pruebas son rápidas, independientes, repetibles, autovalidables y oportunas.
  4. Un ASSERT por prueba: Cada prueba verifica una única condición.
  5. Scope de la prueba: Solo se prueban métodos públicos.

Estas prácticas garantizan que las pruebas sean claras, mantenibles y confiables, lo que es esencial para un desarrollo de software robusto y de alta calidad.

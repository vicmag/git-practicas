Guía de TDD en Java con JUnit, Mockito y Arquitectura Hexagonal

Historia de Usuario

Como administrador de tienda,

Quiero recibir alertas de inventario bajo de los productos

Para que pueda reabastecer el almacén antes de que acabe el producto.

Arquitectura Hexagonal

La arquitectura hexagonal divide la aplicación en tres capas principales:

  • Capa de Dominio: Contiene la lógica de negocio y las entidades.
  • Capa de Aplicación: Orquesta los casos de uso y conecta el dominio con la infraestructura.
  • Capa de Infraestructura: Implementa detalles técnicos como bases de datos, APIs, etc.

Paso 2: Fase Roja (Escribir una Prueba que Falle)

Crea una prueba unitaria para verificar la lógica de alertas de bajo stock.

package com.storemanager.domain.service;

import com.storemanager.domain.model.Product;
import com.storemanager.domain.repository.ProductRepository;
import org.junit.jupiter.api.Test;
import org.mockito.Mockito;

import static org.junit.jupiter.api.Assertions.assertTrue;
import static org.mockito.Mockito.when;

class LowStockAlertServiceTest {

    @Test
    void testCheckLowStock_WhenStockIsLow_ReturnsTrue() {
        // Arrange
        Product product = new Product("1", "Product A", 5, 10); // Stock bajo
        ProductRepository mockRepository = Mockito.mock(ProductRepository.class);
        when(mockRepository.findById("1")).thenReturn(product);

        LowStockAlertService service = new LowStockAlertService(mockRepository);

        // Act
        boolean isLowStock = service.checkLowStock("1");

        // Assert
        assertTrue(isLowStock);
    }
}

Paso 3: Fase Verde (Implementar la Lógica)

Implementa la lógica en la capa de dominio para que la prueba pase.

package com.storemanager.domain.model;

public class Product {
    private String id;
    private String name;
    private int currentStock;
    private int lowStockThreshold;

    public Product(String id, String name, int currentStock, int lowStockThreshold) {
        this.id = id;
        this.name = name;
        this.currentStock = currentStock;
        this.lowStockThreshold = lowStockThreshold;
    }

    public String getId() {
        return id;
    }

    public int getCurrentStock() {
        return currentStock;
    }

    public int getLowStockThreshold() {
        return lowStockThreshold;
    }
}

package com.storemanager.domain.repository;

import com.storemanager.domain.model.Product;

public interface ProductRepository {
    Product findById(String id);
}

package com.storemanager.domain.service;

import com.storemanager.domain.model.Product;
import com.storemanager.domain.repository.ProductRepository;

public class LowStockAlertService {
    private final ProductRepository productRepository;

    public LowStockAlertService(ProductRepository productRepository) {
        this.productRepository = productRepository;
    }

    public boolean checkLowStock(String productId) {
        Product product = productRepository.findById(productId);
        return product.getCurrentStock() <= product.getLowStockThreshold();
    }
}

Paso 4: Refactorización

Mejora el código para que sea más legible y mantenible.

public boolean checkLowStock(String productId) {
    Product product = productRepository.findById(productId);
    return isLowStock(product);
}

private boolean isLowStock(Product product) {
    return product.getCurrentStock() <= product.getLowStockThreshold();
}

Paso 5: Pruebas Adicionales

Agrega más pruebas para cubrir otros escenarios.

@Test
void testCheckLowStock_WhenStockIsNotLow_ReturnsFalse() {
    // Arrange
    Product product = new Product("1", "Product A", 15, 10); // Stock no bajo
    ProductRepository mockRepository = Mockito.mock(ProductRepository.class);
    when(mockRepository.findById("1")).thenReturn(product);

    LowStockAlertService service = new LowStockAlertService(mockRepository);

    // Act
    boolean isLowStock = service.checkLowStock("1");

    // Assert
    assertFalse(isLowStock);
}

@Test
void testCheckLowStock_WhenProductDoesNotExist_ReturnsFalse() {
    // Arrange
    ProductRepository mockRepository = Mockito.mock(ProductRepository.class);
    when(mockRepository.findById("1")).thenReturn(null);

    LowStockAlertService service = new LowStockAlertService(mockRepository);

    // Act
    boolean isLowStock = service.checkLowStock("1");

    // Assert
    assertFalse(isLowStock);
}

Principios SOLID

  • Single Responsibility Principle (SRP): Cada clase tiene una única responsabilidad. Por ejemplo, LowStockAlertService solo se encarga de verificar el stock bajo.
  • Open/Closed Principle (OCP): El código está abierto para extensión pero cerrado para modificación. Por ejemplo, se pueden agregar nuevos tipos de alertas sin modificar el código existente.
  • Liskov Substitution Principle (LSP): Las subclases pueden reemplazar a sus clases base sin alterar el comportamiento. Por ejemplo, cualquier implementación de ProductRepository puede usarse en LowStockAlertService.
  • Interface Segregation Principle (ISP): Las interfaces son específicas y no obligan a implementar métodos innecesarios. Por ejemplo, ProductRepository solo define el método findById.
  • Dependency Inversion Principle (DIP): Las clases de alto nivel no dependen de clases de bajo nivel, sino de abstracciones. Por ejemplo, LowStockAlertService depende de la interfaz ProductRepository, no de una implementación concreta.

Conclusión

Con esta guía, has implementado TDD en Java con JUnit, Mockito y una arquitectura hexagonal para la historia de usuario de alertas de bajo stock. Siguiendo las fases de TDD (Rojo, Verde, Refactorización), has construido una solución robusta y mantenible.

Nota: Asegúrate de ejecutar las pruebas en cada fase con el comando mvn test.

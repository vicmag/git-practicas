Escenario: Bug en la Validación de Stock Negativo

Supongamos que el área de QA reporta un bug: el sistema permite guardar productos con stock negativo, lo que genera inconsistencias en el inventario. Aplicaremos TDD para resolverlo.

Clases Involucradas


// Product.java
public class Product {
    private String id;
    private String name;
    private int stock;

    // Constructor, getters y setters
}

// ProductRepository.java (Interfaz)
public interface ProductRepository {
    void save(Product product);
}

// ProductService.java
public class ProductService {
    private ProductRepository productRepository;

    public ProductService(ProductRepository productRepository) {
        this.productRepository = productRepository;
    }

    public void saveProduct(Product product) {
        productRepository.save(product); // Bug: No valida stock negativo
    }
}

Paso 1: Reporte del Bug por QA

El área de QA identifica que un producto con stock negativo se guarda sin errores:


// Prueba manual de QA
Product product = new Product("1", "Camiseta", -10); // Stock negativo
productService.saveProduct(product); // Se guarda sin validación → BUG

Paso 2: Escribir una Prueba que Falla (Fase Red)

El desarrollador escribe una prueba unitaria que reproduce el bug:


// ProductServiceTest.java
@Test
void whenSaveProductWithNegativeStock_thenThrowException() {
    // Configuración
    ProductRepository mockRepository = Mockito.mock(ProductRepository.class);
    ProductService productService = new ProductService(mockRepository);
    Product invalidProduct = new Product("1", "Camiseta", -10); // Stock negativo

    // Ejecutar y Verificar
    assertThrows(IllegalArgumentException.class, () -> {
        productService.saveProduct(invalidProduct); // Debe lanzar excepción
    });

    // Verificar que NO se llamó a save (opcional)
    verify(mockRepository, never()).save(any());
}

Resultado: La prueba falla porque el sistema permite guardar el producto sin validación.

Paso 3: Implementar la Corrección (Fase Green)

El desarrollador modifica el método saveProduct para validar el stock:


// ProductService.java (corregido)
public void saveProduct(Product product) {
    if (product.getStock() < 0) {
        throw new IllegalArgumentException("El stock no puede ser negativo");
    }
    productRepository.save(product);
}

Resultado: La prueba pasa porque ahora se valida el stock.

Paso 4: Refactorizar (Opcional)

Mejorar la claridad del código sin cambiar su comportamiento:


// ProductService.java (refactorizado)
public void saveProduct(Product product) {
    validateStock(product.getStock());
    productRepository.save(product);
}

private void validateStock(int stock) {
    if (stock < 0) {
        throw new IllegalArgumentException("El stock no puede ser negativo");
    }
}

Paso 5: Validación por QA

El área de QA ejecuta nuevamente la prueba manual y confirma que el bug está resuelto:


Product product = new Product("1", "Camiseta", -10);
productService.saveProduct(product); // Ahora lanza IllegalArgumentException ✅

Conclusión

Gracias a TDD:

  • El bug se reprodujo con una prueba unitaria (Red).
  • Se implementó la validación de stock (Green).
  • Se mejoró la legibilidad del código (Refactor).
  • QA verificó que el problema fue resuelto.

Este flujo garantiza que el bug no vuelva a aparecer en el futuro, ya que la prueba unitaria actuará como un "guardian" del código.
